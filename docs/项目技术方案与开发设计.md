# 会晓 AI 项目技术方案与开发设计

本文档记录当前阶段已经确定的技术栈、开发方向、端侧与服务端边界，以及 Android UI Demo 后续必须补齐的 PRD 覆盖范围。需求以根目录 `会晓AI_PRD_V1.2_APP端最终稿.md` 为最高标准，PRD 未写的功能不作为需求。

## 1. 产品开发方向

会晓 AI 定位为移动端优先的 AI 会议知识与行动平台，不是 PC Web 产品，也不是普通录音转文字工具。

核心方向：

- Android 先行，iOS 后续单独开发。
- Android 技术栈确定后不再替换，后续 iOS 可使用独立技术栈重写。
- App 端优先，本地优先，能放在手机本地的数据优先放本地。
- 服务端只承担手机端无法高效完成或必须共享的能力。
- 当前阶段先完成真实 Android 前端高保真 Demo，再逐步接入后端、ASR、LLM、存储与真实数据。

## 2. 技术栈

### Android App

- 开发语言：Kotlin
- UI 框架：Jetpack Compose
- 设计体系：Material 3 + 自定义会晓 AI 移动端视觉规范
- 架构方向：MVVM
- 状态与异步：ViewModel + Kotlin Coroutines + Flow
- 本地数据库：Room
- 本地文件：App 私有目录 + Android MediaStore
- 后台任务：WorkManager + Foreground Service
- 网络请求：Retrofit 或 Ktor
- 实时通信：WebSocket
- 构建工具：Gradle Wrapper + Android Gradle Plugin

当前 Android 工程位置：

```text
F:\会晓AI\apps\android_app
```

当前 Demo 已可构建：

```text
F:\会晓AI\apps\android_app\app\build\outputs\apk\debug\app-debug.apk
```

### iOS App

iOS 暂不开发，但未来不要求复用 Android 前端。

后续建议：

- Swift
- SwiftUI
- iOS 原生音频、文件、分享、通知、权限能力
- 与 Android 共享服务端接口协议和数据模型

### 服务端

服务端不是 UI 主体，只负责重计算、模型调用、共享和同步能力。

推荐技术：

- Python
- FastAPI
- WebSocket
- 共享远程关系型数据库，当前通过 `HUIXIAO_DATABASE_URL` 显式配置
- Redis
- 对象存储，暂定 OSS
- 后续按需要接入向量数据库或 pgvector 等能力

### ASR 与 AI

ASR：

- 当前模型主线：FunASR runtime 负责实时/离线 ASR，CAMPPlus 负责说话人处理
- 部署位置：服务端
- 负责实时/准实时转写、文件转写、时间戳、说话人处理、置信度等

LLM：

- 部署位置：服务端
- 负责会议摘要、议题、决策、待办、风险、问题、知识库问答
- 具体模型供应商后续再定，App 不绑定具体模型平台

## 3. 目录规划

建议长期目录结构：

```text
F:\会晓AI
├─ apps
│  └─ android_app              # Android App 工程
├─ services
│  ├─ api_server               # App 统一业务接口
│  ├─ asr_service              # FunASR 网关 / 说话人处理
│  └─ ai_server                # 纪要、待办、知识库问答
├─ packages
│  └─ shared_contracts         # 接口协议、数据结构、枚举定义
├─ docs                        # 技术方案、接口文档、设计说明
├─ prototypes                  # HTML 原型历史版本
├─ poc                         # 模型实验和验证
└─ infra                       # 部署脚本、环境配置、对象存储配置
```

历史 POC 目录：

- `Dolphin`：历史 Dolphin 模型源码，不是当前 ASR 主线
- `Dolphin_poc`：历史 ASR 测试工作区
- `Dolphin_poc_env`：当前本机仍复用的 Python 环境命名，但不代表 ASR 服务会加载 Dolphin 模型

这些不应混入 Android App 工程，也不应提交为 App 源码。

## 5. 文件存储策略

原则：能放用户手机本地的长期个人数据，优先放本地。

手机本地保存：

- 用户最终保存的录音
- 本地会议记录
- 本地转写缓存
- 本地导出文件
- 本地缓存文件

服务器或 OSS 保存：

- 需要服务端转写的大音频临时文件
- 云同步开启后的会议音频
- 分享链接关联文件
- 多端访问所需文件
- 服务端模型处理过程中的中间文件

OSS 不是所有音频的默认长期归档位置，只在云同步、分享、服务端处理和多端访问场景使用。

## 7. 当前 Android UI Demo 缺口

当前版本已经具备：

- 首页
- 实时记录页
- 会议详情页
- 待办页
- 知识库页
- 我的页
- 底部导航
- 开始记录弹窗
- 来源核验弹窗
- 分享导出弹窗
- 删除确认弹窗

但当前版本明显不足：

- 页面信息密度不足，仍偏 Demo 骨架。
- 首页没有完整体现会议状态、上传任务、日历会议、待办摘要、项目/客户/标签。
- 实时记录页缺少录音生命周期、权限异常、网络异常、ASR 异常、重点标记、图片标记、实时要点与待办草稿的完整状态。
- 会议详情页缺少编辑态、议题、决策、风险、问题、关键引用、模板化纪要、重新生成、导出格式选择。
- 转写页缺少说话人重命名、批量重命名、文本编辑、置信度、音频定位播放。
- 待办页缺少负责人、截止日期、优先级、状态流转、提醒、来源核验、下次会议回顾。
- 知识库页缺少项目/客户/标签筛选、全文搜索结果、来源会议列表、答案引用结构。
- 我的页缺少账号、个人空间、云同步、权限状态、隐私设置、分享链接管理的完整层级。
- 视觉上仍需提升真实国内办公 App 的完成度，不能停留在列表骨架。

## 8. UI 迭代方向

下一版 Android UI 不应只是“把 HTML 搬到 Compose”，而应按 PRD 做真实产品页。

设计方向：

- 参考国内办公、会议、录音转写类 App 的真实移动端结构。
- 以分组列表、任务状态、时间线、底部操作、详情分段为主体。
- 减少大面积装饰卡片和展示型布局。
- 首页要有真实工作台感，不只是开始按钮和会议列表。
- 会议详情要成为核心复杂页面，覆盖纪要、原文、待办、议题、决策、风险、来源核验、编辑导出。
- 实时记录页要体现录音中的多状态：正在连接、转写中、弱网、暂停、继续、结束、失败后保留录音。
- 待办页要体现行动闭环，而不是简单勾选列表。
- 知识库页要体现“跨会议追问 + 来源引用”，不能只是一个搜索框。

## 9. 开发节奏建议

第一阶段：Android 高保真前端 Demo

- 用假数据完整覆盖 P0 页面与关键 P1 入口。
- 不接后端、不接 ASR、不接 LLM。
- 目标是让上级能判断产品完整度、视觉质量和交互路径。

第二阶段：本地能力接入

- 录音权限
- 本地录音文件
- 本地会议数据
- 本地导出文件
- 分享面板

第三阶段：服务端能力接入

- ASR 服务
- 说话人分离
- AI 纪要
- 待办提取
- 知识库问答
- OSS 临时文件与分享文件

第四阶段：账号、云同步、支付

- 登录
- 个人空间
- 云同步
- 额度
- 支付

## 10. 当前结论

当前技术方向成立，但 UI Demo 还不够。下一步应继续在真实 Android 工程中迭代高保真页面，不再以 HTML 原型作为主要交付物。

下一版 Android UI 的目标不是“能点”，而是“PRD 主要功能在前端都有明确页面、状态和操作入口”。
