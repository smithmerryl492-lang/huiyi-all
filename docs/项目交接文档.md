# 会晓 AI 项目交接文档

更新时间：2026-06-10

## 1. 项目定位

会晓 AI 是移动端优先的 AI 会议知识与行动平台。当前主线是 Android 先行，覆盖会议录音、实时转写、匿名说话人分离、文件导入转写、AI 纪要、待办闭环、来源核验、知识库问答、本地优先保存和云端同步。

需求最高依据是根目录 `会晓AI_PRD_V1.2_APP端最终稿.md`。PRD 未写的功能不应作为当前需求扩张依据。

## 2. 当前架构

```text
Android App
      -> api_server 统一业务接口，本地端口 8080 / 测试服端口 28080
      -> asr_service，端口 8081
          -> FunASR runtime WebSocket，端口 10095
      -> ai_service，端口 8082
      -> 共享远程数据库；所有环境必须通过 HUIXIAO_DATABASE_URL 显式配置，不允许静默回退本机数据库
```

App 只直接对接 `api_server`。ASR、AI、知识库索引、云同步和任务处理由服务端承担。App 端负责录音、文件导入、本地保存、页面交互、待办编辑、分享导出、日历和通知能力。

重要：不要只看文档或 package name 判断环境。2026-06-03 排查确认过，本地直启服务曾因为 `services/.env` 中的 `HUIXIAO_DATABASE_URL` 指向外部 MySQL，而测试服使用 Docker PostgreSQL，导致同一个手机号在两套数据库中生成不同 userId，云同步数据不互通。2026-06-04 已将仓库配置改为：API 缺少 `HUIXIAO_DATABASE_URL` 时直接启动失败，测试服和本地 Docker 不再创建 Postgres 容器，staging 默认地址指向测试服。任何云同步问题仍必须先核对两端 health 的 `database`、`data_dir` 和 App BuildConfig 地址。

## 3. 主要目录

- `apps/android_app`：Android Kotlin + Jetpack Compose 工程。
- `services/api_server`：FastAPI 统一后端入口，处理上传、任务、结果、同步、知识库和实时 WS 代理。
- `services/asr_service`：ASR 服务，当前只作为 FunASR runtime 的文件/实时转写网关和说话人修复辅助。
- `services/ai_service`：OpenAI 兼容 LLM/embedding 服务，生成纪要、待办、风险和知识库答案。
- `services/funasr_runtime`：FunASR WebSocket runtime，本地启动脚本在 `services/funasr_runtime/start_runtime.ps1`。
- `packages/contracts`：共享 schema。当前仍以 `api_server` OpenAPI 为准。
- `infra`：Docker Compose 骨架。
- `test_assets`：会议音频、说话人分离测试音频、脚本和 timeline。
- `Dolphin`、`Dolphin_poc`、`Dolphin_poc_env`：历史 POC 和本机 Python 环境，不是当前 ASR 主线模型来源。
- `tools/ffmpeg`：Windows/Linux ffmpeg 和 ffprobe 二进制。
- `ui_refs`：竞品/历史 UI 参考图。

## 4. Android 当前状态

入口：

- `apps/android_app/app/src/main/java/com/huixiaoai/app/MainActivity.kt`
- `apps/android_app/app/src/main/java/com/huixiaoai/app/HuixiaoApp.kt`
- `apps/android_app/app/src/main/java/com/huixiaoai/app/state/HuixiaoAppState.kt`

页面和弹窗：

- `screens/Screens.kt`：首页、预约、导入、录音、生成中、会议详情、待办、知识库、我的等页面。
- `sheets/Sheets.kt`：录音确认、预约编辑、来源核验、分享导出、待办详情、删除确认、隐私说明等弹窗。
- `components/Components.kt` 和 `ui/Theme.kt`：轻量组件与颜色。

重要实现：

- 录音：`recording/AndroidAudioRecorder.kt` 使用 `AudioRecord` 采集 16kHz mono PCM，同时写本地 WAV。
- 前台录音服务：`recording/RecordingForegroundService.kt`，通知栏持续显示录音状态。
- 实时转写：`data/LiveTranscriptionClient.kt`，WebSocket 发送 PCM 帧，接收 `transcript.partial/final`。
- 文件导入：`fileimport/LocalFileImporter.kt`，支持 mp3/m4a/wav/aac/mp4/mov，最大 500MB。
- 来源音频播放：`audio/AudioSegmentPlayer.kt`。播放层只接收本机文件路径并用 `MediaPlayer.setDataSource(File(path).absolutePath)` 播放，不直接流式读取远端音频。
- 云端会议音频缓存：`data/CloudAudioCache.kt` 会把 `/tasks/{remoteTaskId}/audio` 下载到 App 沙盒 `files/cloud_audio/`，再把 `MeetingTask.localFilePath` / `Meeting.sourceFilePath` 指向该沙盒文件。录音文件在 `files/recordings/`，导入文件在 `files/imports/`，三者都不应依赖公共手机存储原始地址。
- 验证云端音频拉取时必须先确认 `files/cloud_audio/` 是否为空。若同一个 `remoteTaskId` 已有旧缓存，`CloudAudioCache.cachedAudioPath()` 会直接复用本地缓存，不会重新访问远端下载接口，可能把“远端拉取失败”误判为播放正常。
- 日历/待办提醒：`calendar/CalendarEventWriter.kt`、`notifications/ScheduleStatusNotifier.kt`。

当前本地存储不是 Room，而是 SharedPreferences JSON：

- `LocalTaskStore.kt`
- `LocalMeetingResultStore.kt`
- `LocalScheduleStore.kt`
- `LocalKnowledgeStore.kt`
- `CloudSyncOperationStore.kt`
- `CloudAccountModels.kt`

ASR 模型所有权和危险遗留处理：

- 2026-06-04 已移除 `asr_service/app/inference/dolphin_engine.py`，ASR 服务不再导入旧 Dolphin ASR 入口，也不再暴露 Dolphin 模型路径健康字段。
- 离线 ASR、在线 ASR、VAD、PUNC 和 runtime 内 CAMPPlus 只由 `funasr_runtime` 进程加载。不要在 `asr_service` 内恢复第二套离线 ASR 模型，否则可能重现测试机内存峰值过高、容器 137/OOM、连接重置等问题。
- `deploy/funasr_wss_server.py` 的离线模型固定为 `paraformer-zh`，不再开放未使用的 `--asr_model` 覆盖参数。上下文模型、SenseVoice、Whisper/faster-whisper 等替换只能作为隔离实验，不是上线前主线。
- `funasr_live_gateway.py` 中旧实时音频预处理代码已删除。该类历史上曾接入后又被移除，不能再把它当成“忘记启用”的优化；恢复前必须用 CER、漏字、延迟、CPU 和多人会议样本做 A/B。
- 热词当前禁用。热词可能提升专有名词，也可能把普通词误偏成热词，改变转写原文并影响纪要、知识库和 AI 问答。只有热词命中率、误触发率和下游 QA 结果都过线后，才能单独开实验开关。
- 文件转写的 ASR service -> FunASR runtime 传输默认使用 JSON 元信息 + PCM binary frame，旧 `file_pcm_transcribe` + `audio_b64` 仅作为兼容入口保留。不要把 base64 JSON 大载荷恢复为默认性能路径；相关学习和基线记录见 `docs/ASR性能准确率优化学习记录.md`。

`ai_service`：

- 启动入口：`services/ai_service/app/main.py`
- OpenAI 兼容客户端：`services/ai_service/app/clients/openai_compatible.py`
- 配置：`services/ai_service/app/core/config.py`

当前本机已见到的监听端口包括 `8080`、`8081`、`8082`、`10095`、`55432`。

本地配置警告：

- `services/start_local_services.ps1` 会先读取 `services/.env` 并注入当前进程环境。
- `services/.env` 内含中文短信签名（如 `HUIXIAO_TENCENT_SMS_SIGN_NAME=江西六角星`）时，PowerShell 读取必须显式使用 UTF-8。2026-06-06 曾因脚本未指定 `Get-Content -Encoding UTF8`，文件和腾讯云后台配置均正确，但 API 进程实际拿到乱码签名，导致本地短信登录返回 `signature format is incorrect or signature is not approved`。排查时要同时验证“文件内容”和“运行进程实际发送参数”，不能只看 `.env`。
- 2026-06-10 本地短信模板已从鲲穹翻译旧模板 `2647439` 切到会晓模板 `2660883`，并且因为当前会晓只有一个腾讯云短信模板，`HUIXIAO_SMS_TEMPLATE_LOGIN_ID`、`HUIXIAO_SMS_TEMPLATE_CHANGE_PHONE_ID`、`HUIXIAO_SMS_TEMPLATE_CHANGE_PASSWORD_ID` 三项都填 `2660883`。`.gitea/workflows/deploy-test.yml` 会在选中测试服 env 后把三项模板 ID 归一为 `2660883`，避免 Gitea Secret 中旧 `2647439` 继续生效。每次部署仍必须看日志审计确认实际写入值。
- 2026-06-12 起，短信配置改为“单模板 ID 为主、旧三场景字段兼容”的模式：新增 `HUIXIAO_SMS_TEMPLATE_ID` 作为唯一模板来源，`login / change_phone / change_password` 三个旧字段只做兼容回退，三者若存在必须全部与统一 ID 一致。这样可以避免某个场景漏配或恢复旧模板后只影响单一路径。
- 2026-06-12 起，数据库、AI、ASR、声纹和处理卡死兜底都增加了可配置超时：`HUIXIAO_DB_CONNECT_TIMEOUT_SECONDS`、`HUIXIAO_DB_READ_TIMEOUT_SECONDS`、`HUIXIAO_DB_WRITE_TIMEOUT_SECONDS`、`HUIXIAO_ASR_VOICEPRINT_TIMEOUT_*`、`HUIXIAO_ASR_VOICEPRINT_ENROLL_TIMEOUT_SECONDS`、`HUIXIAO_AI_EMBEDDING_TIMEOUT_SECONDS`、`HUIXIAO_AI_KNOWLEDGE_PLAN_TIMEOUT_SECONDS`、`HUIXIAO_AI_KNOWLEDGE_ANSWER_TIMEOUT_SECONDS`、`HUIXIAO_STALE_PROCESSING_TIMEOUT_SECONDS`。本地和测试服都要一起核对这些项，不能只看主模型 key。
- `services/.env` 必须存在 `HUIXIAO_DATABASE_URL`，并且应指向共享远程数据库。脚本不再回退本地 PostgreSQL。
- 启动后必须执行健康检查确认有效数据库：

```powershell
Invoke-RestMethod -Uri 'http://127.0.0.1:8080/api/v1/health' | ConvertTo-Json -Depth 4
```

如果本地和测试服 health 的 `database` 不一致，或登录返回的 userId/`bootstrap` 数量不一致，两端就不是同一个云环境。同手机号、同账号也不能说明云数据互通。

## 10. 配置危机复盘：本地 MySQL 与测试服 PostgreSQL 分叉

2026-06-03 云同步问题排查中出现过一次严重配置误判：本地测试正常，真机连接测试服后提示“云端数据已同步”或“拉取云端数据失败”，但云端会议没有出现在 App 中，新手机/新安装也复现。

最终确认不是手机、账号、APK cleartext、网络权限或包名问题，而是两端有效数据库不同：

- 本地直启 API：`database=mysql`，数据目录为 `F:\会晓AI\services\api_server\data`。
- 测试服 API：`database=postgresql`，数据目录为 `/data`。
- 同手机号 `19914596080` 在两套数据库中对应不同 `userId`；`/sync/bootstrap` 按登录 token 中的 `userId` 查数据，不按手机号跨库合并。
- 当时本地 MySQL 中该手机号有 7 个 tasks / 4 个 results，测试服 PostgreSQL 中该手机号有 4 个 tasks / 1 个 result。两端返回不同数据是配置导致的正常结果，不是客户端没有拉取。

2026-06-04 已做配置层修正：测试服 Compose 不再定义 `postgres` 服务，API 直接读取 `HUIXIAO_DATABASE_URL`；本地脚本、测试服 Compose、本地 Docker Compose 和 API 代码都要求显式配置共享远程数据库。测试服容器已关闭，当前不允许开启，未进行运行时验证；下次开槽后必须先确认 health 不再显示 `database=postgresql`。

以后遇到任何“本地正常、测试服/真机云同步不一致”的问题，必须先执行以下检查：

```powershell
Invoke-RestMethod -Uri 'http://127.0.0.1:8080/api/v1/health' | ConvertTo-Json -Depth 4
Invoke-RestMethod -Uri 'http://43.154.197.96:28080/api/v1/health' | ConvertTo-Json -Depth 4
```

必须逐项比对：

- `database`：不同即不是同一片云；不要继续猜设备、账号或 APK。
- `data_dir`：不同代表文件存储不互通；当前没有 OSS/COS/S3 统一对象存储。
- `asr_service_url`、`asr_live_ws_url`、`ai_service_url`：本地应是 `127.0.0.1`，测试服应是 Docker 内网服务名；这类差异本身正常，但处理结果可能不同。
- App BuildConfig：`staging` flavor 不能证明后端地址，必须确认 API 是 `http://43.154.197.96:28080/api/v1` 还是 `http://10.0.2.2:8080/api/v1`。
- Auth：当前 legacy `?user_id=` 已关闭，两端未登录访问 `/sync/bootstrap` 都应返回 401。云同步必须走短信登录 token。

配置项风险级别：

- 致命：`HUIXIAO_DATABASE_URL`、旧本地数据库兜底配置、实际 DB volume、账号 userId 映射。
- 高风险：`HUIXIAO_DATA_DIR`、Android `HUIXIAO_STAGING_API_BASE_URL` / `HUIXIAO_STAGING_LIVE_WS_URL`、CI `.env` 优先级、`HUIXIAO_AUTH_TOKEN_SECRET`。
- 结果一致性风险：`HUIXIAO_ASR_MAX_SPEAKERS`、FunASR 并发/阈值参数、`HUIXIAO_LLM_TIMEOUT_SECONDS`。
- 当前应保持一致：短信模板/签名、LLM base/model/embedding、LLM timeout/temperature、Auth TTL、上传大小限制、ASR max speakers、FunASR 并发参数。前提是 Gitea Secret、服务器本地私有 `.env` 和实际容器环境没有漂移；仓库不再保存真实 `.env`。

排障纪律：

- 不要再用“同一手机号/同一真机/同一 staging flavor”替代配置核验。
- 不要把“未上传会议=0”当成云同步成功；它只表示本地 finished 且未 synced 的数量。
- “云端数据已同步”目前只说明相关请求没有抛异常，不等于实际新增了会议。
- 任何修复前先记录两端 health、登录返回的 userId、bootstrap task/result 数量，再判断是否需要迁移数据或统一 DB。

## 11. AI 模型与知识库向量切换记录

2026-06-05 已将 LLM/embedding 主线切到阿里云百炼 DashScope OpenAI 兼容接口：

- `HUIXIAO_LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1`
- `HUIXIAO_LLM_MODEL=qwen3.7-plus`
- `HUIXIAO_EMBEDDING_MODEL=text-embedding-v4`
- `HUIXIAO_EMBEDDING_DIMENSIONS=1024`

模型选择原则：上线前优先选稳定、中文会议总结能力强、支持 OpenAI 兼容调用和结构化输出、成本可控的模型。当前选 `qwen3.7-plus`，暂不选更重的 max 档，避免上线前引入过高成本和延迟风险。

纪要生成性能纪律：

- 会议纪要是基于转写内容的结构化摘要/议题/待办/风险抽取，默认必须显式关闭模型 thinking：`HUIXIAO_MINUTES_ENABLE_THINKING=false`。不要依赖 `qwen3.7-plus` 的模型默认值，否则会出现 `enable_thinking=None`、reasoning tokens 很高、LLM 阶段耗时被放大的问题。
- 长会议超时必须按完整处理链路一起核：App 重生成纪要请求、API Server 等 ASR、ASR Service 等 FunASR runtime、API Server 等 AI Service、AI Service 等上游 LLM。不能只改 LLM，否则 30 分钟到 2 小时会议仍可能先死在 API->ASR 固定超时。
- 纪要 LLM 和文件 ASR 超时按会议时长动态放宽，只用于避免卡死，不用于限制正常慢请求。当前不要把长会议分段生成纪要作为上线前默认方案；分段会引入跨段上下文丢失和待办/风险归并不准的风险。
- 当前长会议超时策略：`HUIXIAO_ASR_FILE_TIMEOUT_SECONDS=32400` 作为 API->ASR 外层保险；ASR runtime 使用 `HUIXIAO_FILE_ASR_RUNTIME_TIMEOUT_BUFFER_SECONDS + 音频秒数 * HUIXIAO_FILE_ASR_RUNTIME_TIMEOUT_PER_AUDIO_SECOND`，封顶 `28800` 秒；AI 服务等上游 LLM 封顶 `18000` 秒，API->AI 外层封顶 `21600` 秒，App 侧单次纪要重处理读超时已放宽到 7 小时。后续若支持 2 小时以上会议，应先实测 ASR/LLM 耗时再调整这些上限。
- 纪要链路和知识库问答链路的 thinking 策略不同。知识库可对“原因/建议/分析/复盘”等问题使用小预算 thinking；会议纪要默认关闭，除非有基线 A/B 证明准确率收益大于延迟和超时风险。

密钥纪律：

- 真实阿里云 key 只允许放在本机被 `.gitignore` 忽略的 env、服务器本地 env 或 Gitea Secret，不要写进可提交文件。
- 历史白山/EdgeFn 配置不允许继续使用。提交前必须搜索 `api.edgefn.net`、`DeepSeek-V3.2`、`BAAI/bge-m3`，确认只剩非相关历史文字或已处理。
- `deploy/deploy.sh` 会拒绝非阿里云 base URL、旧模型、旧 embedding、占位 key 或旧 ASR/Dolphin 环境变量，避免误部署。
- `deploy/deploy.sh` 的短信校验已经改成优先核验 `HUIXIAO_SMS_TEMPLATE_ID`，旧三场景字段仅用于兼容。后续如果只改一个短信场景模板，脚本应直接拒绝。

知识库向量链路：

- 旧代码虽然有 `embedding_json` 字段，但索引时实际写入空向量，查询时也丢弃向量。切换阿里云后，`index_meeting_result()` 已改为调用 embedding 服务写入真实向量，查询排序会生成问题向量参与余弦相似度。
- embedding 失败不应打断会议处理：索引失败时降级为空向量，知识库仍走结构化/关键词/实体/时间排序。
- 向量模型切换后不要直接清空会议结果。需要清理旧向量时，只重建 `knowledge_chunks` 索引层：

```powershell
cd F:\会晓AI\services\api_server

& '..\.venv\Scripts\python.exe' 'tools\rebuild_knowledge_index.py'
```

该脚本会删除并重建 `knowledge_chunks`，不会删除 `tasks`、`results`、`files` 或上传文件。测试服/生产库必须在确认执行窗口后再跑。

## 12. 知识库风险查询事故复盘

2026-06-05 测试反馈：会议摘要中已经包含风险，但知识库提问“最近有哪些风险？”返回“没有找到相关风险”。本地复现确认这不是手机网络、部署配置或测试服旧包单独导致的问题，而是 `api_server` 知识库回答链路的代码漏洞。

已确认的故障链路：

- App 会把会议结果里的 `risks` 建成本地 `chunk_type=risk` 知识片段。
- 服务端查询计划能正确识别 `intent=risk_lookup`、`content_types=["risk"]`。
- `_filter_chunks_by_time`、`_rank_chunks_by_plan` 和 `_select_sources` 已经命中风险来源。
- 旧版 `_structured_answer()` 只处理会议、参会人和待办，没有 `risk_lookup` 分支。
- 命中的风险来源继续交给 `ai_service` 兜底回答后，AI 可能返回“没有明确依据”。
- `knowledge.py` 主流程检测到无依据回答后会清空 `sources`，最终表现为“最近没有找到相关风险”。

2026-06-05 已修复：

- 文件：`services/api_server/app/services/knowledge.py`
- 新增 `risk_lookup` 结构化回答分支。
- 新增 `_risk_records()`，只从 `chunk_type=risk` 的片段生成风险列表。
- 新增 `_split_risk_text()`，按“标题。描述。建议/来源”格式拆分展示，过滤纯“依据：”来源字段。
- 风险查询现在与会议/待办一样，已命中结构化来源时直接返回 `model=structured`，不再让 AI 兜底把已命中内容误判成无证据。

验证命令：

```powershell
& 'F:\会晓AI\services\.venv\Scripts\python.exe' -m py_compile services/api_server/app/services/knowledge.py
```

固定复现脚本已验证：同一条 `LocalKnowledgeSource(chunk_type='risk')` 在提问“最近有哪些风险？”时，返回 `最近7天有1项风险...`，`sources` 保留 1 条，`model=structured`。

后续排查纪律：

- 会议摘要/详情中有风险，但知识库问不到风险时，先检查风险 chunk 是否已生成、查询计划是否是 `risk_lookup`、结构化回答是否返回，不要直接判断为部署或网络问题。
- 对会议列表、参会人、待办、风险这类确定性聚合问题，优先补结构化回答，不要完全依赖 LLM 兜底。
- 如果后续新增决策、议题、摘要等确定性聚合问法，也要补对应结构化分支和最小复现，避免“来源已命中但回答说无结果”的同类事故。

### 12.1 知识库“本周我的待办”查询复盘

2026-06-06 测试反馈：常用按钮“本周待办”提问“本周我有哪些待办？”时，回答退化成所有待办；单独问“我有哪些待办”又能正常按当前用户过滤。该问题不是模型调教或提示词问题，本轮没有修改模型、模型参数、系统提示词或阿里云模型配置。

已确认根因：

- 旧链路把“我”的解析交给 AI 查询计划兜底，时间词插入后 `participant_terms` 可能为空。
- `todo_lookup` 的负责人约束为空时，结构化待办回答会退化为所有待办。
- 待办时间范围应按“截止时间”过滤；不能把“本周待办”混同为“本周创建的会议里的待办”。
- 客户端存在未使用的 `identityAwareQuestion` 变量，看起来像把身份拼进问题，实际请求仍只传原始问题和 `user_name`，容易误导排查。

2026-06-06 已修复：

- 文件：`services/api_server/app/services/knowledge.py`
- 自我指代的参会/待办结构化查询直接使用客户端传入的 `user_name`，不再依赖 LLM 猜“我”是谁。
- 查询计划统一归一化原问句中的显式时间范围，避免 AI plan 和本地识别结果漂移。
- `todo_lookup` 只使用 `chunk_type=todo`，并按待办 `截止时间` 判断“今天/本周/上周/最近 N 天”等范围。
- 对“我”的待办查询，如果没有可靠身份，不允许退化成全量待办。
- 删除 Android 端未使用的 `identityAwareQuestion` 和 `withProfileIdentity()`，避免后续误判身份增强发生在客户端。

新增回归测试：

- 文件：`services/api_server/tests/test_knowledge_self_todo.py`
- 覆盖“本周我有哪些待办”“无 user_name 时不能返回所有待办”“本周张三有哪些待办”“本周我参加了哪些会议”“我是谁不应被强制转成结构化查询”。

验证命令：


后续排查纪律：

- 凡是“我/本人/自己”的待办、参会类聚合问题，必须先确认请求中 `user_name` 是否进入服务端查询计划，再看检索结果。
- 不要通过给 LLM 提示词继续打补丁来解决确定性聚合问题；这类问题应在 `api_server` 查询计划和结构化回答层处理。
- 常用按钮新增或改文案后，要补一条结构化查询回归用例，避免按钮文案一变就绕开原有规则。
